# АНАЛИЗ И РАСПРЕДЕЛЕНИЕ RPC НАГРУЗКИ (LOOKHOOK)

## 1. ДОСТУПНЫЕ ПОСТАВЩИКИ (Провайдеры)
На основе .env файла в проекте настроено 3 независимых провайдера. Сейчас активен **Ankr** (высокая скорость).
У вас есть 7 уникальных ключей Ankr, закрепленных за разными почтами.

---

## 2. АНАЛИЗ ТЕКУЩЕГО ИСПОЛЬЗОВАНИЯ (Где "горит" ресурс)

*   **Key 2 (MINING):** ПЕРЕГРУЖЕН. Используется для всего блока Mining: инвентарь, балансы USDC, баланс пула наград, проверка стейкинга каждой карточки.
*   **Key 4 (OG/SOCIAL):** СРЕДНЯЯ НАГРУЗКА. Один ключ делится между OG-профилем и всеми социальными наградами (X, TG, YT) + Аирдроп.
*   **Key 7 (STAKE):** ВЫСОКАЯ НАГРУЗКА. Используется для всех расчетов стейкинга и клейма наград.
*   **Key 3 (TIPS):** ПОЧТИ НЕ ИСПОЛЬЗУЕТСЯ. Используется только для "BuyMeACoffee", который вызывается крайне редко.

---

## 3. ПЛАН РАВНОМЕРНОГО РАСПРЕДЕЛЕНИЯ (UTILIZATION)

Чтобы задействовать Key 3, 4 и 7 эффективнее и максимально разгрузить Key 2, предлагаю следующий маппинг:

| Модуль / Задача | Текущий Клиент | Рекомендуемый Ключ | Почему? |
| :--- | :--- | :--- | :--- |
| **Mining (Инвентарь)** | Key 2 | **Key 2** | Оставляем основной тяжелый multicall здесь. |
| **Mining (USDC Balance)**| Key 2 | **Key 3 (TIPS)** | Разгружаем Key 2. Key 3 почти свободен, идеально для балансов. |
| **Mining (Pool Rewards)**| Key 2 | **Key 7 (STAKE)** | Логически верно: информация о пулах наград должна идти через Stake-ключ. |
| **Airdrop / Achievements**| Key 4 | **Key 3 (TIPS)** | Еще больше нагружаем Key 3, разгружая Key 4. |
| **OG Registry** | Key 4 | **Key 4** | Оставляем для работы с профилями. |
| **Social Rewards (X)** | Key 4 | **Key 1 (GM)** | Модуль X теперь легкий (агрегатор), его можно "поселить" к базовым функциям GM. |

---

## 4. КАК ЭТО ЗАДЕЙСТВОВАТЬ В КОДЕ? (Технический план)

1.  **В `src/partials/Mining.tsx`**:
    *   Заменить `miningPublicClient` на `tipsPublicClient` для запроса `usdcBalance`.
    *   Заменить `miningPublicClient` на `stakePublicClient` для запроса `totalRewards`.
    *   *Результат: При открытии Mining запросы полетят параллельно по ТРЕМ разным каналам (2, 3 и 7), что ускорит загрузку и снимет лимиты с Key 2.*

2.  **В `src/views/Airdrop.tsx`**:
    *   Заменить `xPublicClient` на `tipsPublicClient`.
    *   *Результат: Аирдроп не будет мешать работе социальных наград (Key 4).*

3.  **В `src/hooks/useSocialRewardsAggregator.ts`**:
    *   Заменить `xPublicClient` на `publicClient` (Key 1).
    *   *Результат: Полная изоляция модулей друг от друга.*

---
**Итог**: Мы задействуем простаивающий Key 3 (Tips) и мощный Key 7 (Stake) для подстраховки Mining, а Key 4 освобождаем для чистой работы с OG-профилями. Это сделает работу приложения максимально стабильной.
